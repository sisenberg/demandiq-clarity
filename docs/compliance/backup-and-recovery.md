# CasualtyIQ — Backup & Recovery

> **Status**: Readiness hardening baseline. No automated recovery testing exists yet.

## 1. What Must Be Recoverable

### Tier 1 — Critical (Data loss = service failure)

| Asset | Location | Recovery Source |
|---|---|---|
| Case data (all tables) | Postgres database | Platform automatic backups |
| Uploaded documents | `case-documents` storage bucket | Platform storage redundancy |
| Generated artifacts | `derived-artifacts` storage bucket | Platform storage redundancy |
| User accounts & auth | `auth.users` (Supabase-managed) | Platform automatic backups |
| Tenant/profile data | `tenants`, `profiles`, `user_roles` | Platform automatic backups |

### Tier 2 — Important (Data loss = rework required)

| Asset | Location | Recovery Source |
|---|---|---|
| OCR extracted text | `case_documents.extracted_text`, `document_pages` | Re-extractable from source documents |
| Document classifications | `document_type_suggestions`, `document_metadata_extractions` | Re-extractable via `classify-document` |
| Entity clusters | `entity_clusters`, `entity_cluster_members` | Re-extractable via `normalize-entities` |
| Chronology candidates | `chronology_event_candidates`, `chronology_evidence_links` | Re-extractable via `generate-chronology` |

### Tier 3 — Operational (Data loss = inconvenience)

| Asset | Location | Recovery Source |
|---|---|---|
| Audit events | `audit_events` | Platform backup only; not re-generatable |
| Job history | `intake_jobs`, `jobs` | Platform backup only |
| Module completion state | `module_completions`, `module_completion_snapshots` | Platform backup only |

## 2. RPO / RTO Assumptions

| Metric | Target | Current Capability | Gap |
|---|---|---|---|
| **RPO** (Recovery Point Objective) | ≤ 24 hours | Platform daily backups (assumed) | No verified backup frequency; no point-in-time recovery confirmation |
| **RTO** (Recovery Time Objective) | ≤ 4 hours | Platform restore from backup | No tested restore procedure; no runbook |

| **Backup retention** | 30 days minimum | Platform-managed | Retention period depends on Supabase tier |
| **Cross-region backup** | Desired for DR | Not confirmed | Depends on platform tier |

## 3. Recovery Procedures (Current State)

### Database Recovery

1. **Source**: Supabase automatic backups
2. **Method**: Platform-managed point-in-time recovery (if available on tier) or daily snapshot restore
3. **Scope**: Full database restore (all tenants)
4. **Tested**: ❌ No recovery test performed

### Storage Recovery

1. **Source**: Platform storage redundancy (S3-backed)
2. **Method**: Objects are durable by platform design
3. **Scope**: Individual objects or full bucket
4. **Tested**: ❌ No recovery test performed

### Derived Data Recovery

Tier 2 data (OCR text, classifications, entities, chronology) can be regenerated by re-running the intake pipeline:

1. Re-invoke `process-document` for each document → regenerates `document_pages` and `extracted_text`
2. Re-invoke `classify-document` → regenerates type suggestions and metadata extractions
3. Re-invoke `normalize-entities` → regenerates entity clusters
4. Re-invoke `generate-chronology` → regenerates timeline candidates

**Note**: User corrections (accepted types, edited metadata, chronology edits) stored in the database would need to be recovered from backup — they cannot be regenerated.
